Version control
=================

Server-client model: (ex SVN, CVS)
	Original style, still widely used (e.g. CVS, Subversion) One central repository on server.
	Developers‚Äô workflow (simplified!):
		‚Ä¢ Check out a working copy,
		‚Ä¢ Make changes, test and debug,
		‚Ä¢ Check in (commit) changes to repository (with comments).
		This creates new version number.
		‚Ä¢ Run an update on working copy to bring in others‚Äô
		changes.

	The system keeps track of diffs from one version to the next (and info on who made the changes, when, etc.)
	A changeset is a collection of diffs from one commit.

	Only the server has the full history. The working copy has:
		‚Ä¢ Latest version from repository (from last checkout, commit, or update)
		‚Ä¢ Your local changes that are not yet committed. Note:
		‚Ä¢ You can retrieve older versions from the server.
		‚Ä¢ Can only commit or update when connected to server.
		‚Ä¢ When you commit, it will be seen by anyone else who does an update from the repository.

	Often there are trunk and branches subdirectories.


Distributed version control:

	Git uses a distributed model:
		- git commit commits to your clone‚Äôs .git directory.
		‚Ä¢ git push sends your recent changesets to another clone by default: the one you cloned from (e.g. bitbucket),
		but you can push to any other clone (with write permission).
		‚Ä¢ git fetch pulls changesets from another clone
		by default: the one you cloned from (e.g. bitbucket)
		‚Ä¢ git merge applies changesets to your working copy



	Advantages of distributed model:
		‚Ä¢ You can commit changes, revert to earlier versions, examine history, etc. without being connected to server.
		‚Ä¢ Also without affecting anyone else‚Äôs version if you‚Äôre working collaboratively. Can commit often while debugging.
		- No problem if server dies, every clone has full history.


		For collaboration will still need to push or fetch changes eventually and git merge may become more complicated.
